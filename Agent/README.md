# Document Parsing Architecture

Этот `README.md` подробно описывает архитектуру и логику двух основных файлов проекта:

- `code/document_parser_agent.py`
- `code/example_parsing_1.py`

Оба файла решают похожую задачу: взять изображение документа, отправить его в мультимодальную модель, получить структурированные данные и сохранить JSON.  
Но уровень сложности у них разный:

- `code/document_parser_agent.py` — базовая, более простая версия.
- `code/example_parsing_1.py` — расширенная production-like версия с двухшаговым пайплайном, дополнительной валидацией и повторными попытками исправления JSON.

---

## 1. Общая идея проекта

Проект строится вокруг одной практической задачи:

1. взять фото документа;
2. распознать текст;
3. превратить текст в строго структурированный JSON;
4. проверить, что JSON действительно валиден и не потерял важные данные;
5. сохранить итог рядом с изображением.

Главная проблема таких пайплайнов в том, что LLM часто:

- отдают JSON не по схеме;
- пропускают строки таблицы;
- путают числовые форматы (`"1 250,50"` вместо `1250.50`);
- возвращают “почти JSON”, но с мусором вокруг;
- в длинных документах обрезают список позиций.

Именно поэтому в проекте есть не только генерация JSON, но и отдельный слой **контроля качества**.

## 2. Архитектура `code/document_parser_agent.py`

`code/document_parser_agent.py` — это базовая версия пайплайна.

### 2.1. Что делает этот файл

Он:

1. выбирает изображение;
2. кодирует его в base64;
3. отправляет картинку в одну модель;
4. просит модель сразу вернуть JSON;
5. пытается распарсить ответ;
6. валидирует по Pydantic-схеме;
7. сохраняет JSON в файл.

### 2.2. Основные блоки

#### Авто-перезапуск в `.venv`

Функции:

- `_running_in_venv()`
- `_maybe_reexec_into_local_venv()`

Нужны для того, чтобы скрипт всегда запускался тем Python-интерпретатором, где стоят зависимости проекта.

#### Получение API-ключа

Функции:

- `_load_api_key_from_dotenv()`
- `_get_groq_api_key()`

Ищут ключ в:

1. `GROQ_API_KEY` из окружения;
2. `.env` рядом с файлом.

#### Схема данных

Используются:

- `InvoiceItem`
- `DocumentSchema`

Это Pydantic-модели, которые описывают итоговый формат JSON.

#### Состояние графа

Используется `AgentState`, где лежат:

- `image_bytes`
- `parsed_json`
- `error`

Это минимальное состояние для очень простого графа.

#### Логика парсинга

Главная функция:

- `parse_document_node(state)`

Она:

1. собирает промпт;
2. вызывает модель Groq;
3. получает поток текста;
4. собирает финальный ответ;
5. запускает `parse_ai_json()`;
6. либо возвращает `parsed_json`, либо кладёт ошибку в `error`.

#### Сборка графа

Граф в этом файле очень простой:

`START -> parser_node -> END`

То есть это фактически **один узел**, обёрнутый в LangGraph.

### 2.3. Ограничения этой версии

Этот файл хорош как простой MVP, но у него есть ограничения:

- одна модель делает всё сразу;
- нет отдельного OCR-этапа;
- нет цикла самоисправления;
- нет повторной коррекции JSON;
- качество сильно зависит от того, насколько хорошо одна модель одновременно видит изображение и структурирует ответ.

## 3. Архитектура `code/example_parsing_1.py`

`code/example_parsing_1.py` — это расширенный пайплайн, где задача разделена на **два этапа**.

### 3.1. Зачем нужен второй файл

Вместо “одна модель делает всё” здесь используется более надёжная схема:

1. **OCR-модель** сначала читает документ как текст;
2. **JSON-модель** получает уже текст и строит структурированный JSON;
3. отдельный узел проверяет качество результата;
4. если JSON плохой — модель получает обратную связь и делает новую попытку.

Это гораздо устойчивее на длинных накладных, ТТН и отчётах.

---

## 4. Подробно по `example_parsing_1.py`

### 4.1. Основные модели данных

В файле используются три основные Pydantic-модели:

#### `InvoiceItem`

Одна строка товарной таблицы:

- `name`
- `unit_of_measurement`
- `quantity`
- `price`
- `amount_without_vat`
- `vat_rate`
- `vat_amount`
- `amount_with_vat`
- `notes`

#### `PeriodRecord`

Одна запись для отчётов/ведомостей:

- `document_number`
- `document_date`
- `quantity`

#### `DocumentSchema`

Корневой итоговый JSON:

- тип документа;
- номер;
- дата;
- поставщик;
- покупатель;
- суммы;
- список `items`;
- поля для отчётов (`purchase_form`, `opening_balance`, `expense`, `closing_balance`, `periods`);
- `confidence_score`.

---

### 4.2. Нормализация чисел до валидации

Очень важный блок:

- `_parse_float()`
- `@field_validator(..., mode="before")`

#### Зачем это нужно

OCR и LLM часто возвращают числа в “человеческом” формате:

- `"1 250,50"`
- `"10 000,00"`
- `"1,250.50"`

Pydantic ожидает `float`, а не строку.  
Без предобработки это приводило бы к `ValidationError`, даже если смысл числа распознан правильно.

#### Что делает `_parse_float()`

Функция:

- убирает пробелы и спецпробелы;
- оставляет только цифры, `,`, `.`, `-`;
- определяет десятичный разделитель;
- приводит строку к `float`.

#### Почему это сделано именно в моделях

Потому что это правильное место с точки зрения архитектуры:

- бизнес-схема сама знает, как приводить свои поля;
- логика не размазывается по узлам;
- любой JSON, который проходит через `DocumentSchema`, автоматически получает одинаковую нормализацию.

---

### 4.3. Состояние графа

`AgentState` в `example_parsing_1.py` богаче, чем в базовом файле.

Он хранит:

- `image_bytes` — изображение в base64;
- `ocr_text` — результат OCR-узла;
- `raw_json_text` — сырой ответ второй модели;
- `parsed_json` — уже прошедший проверку JSON;
- `validation_error` — причина невалидности;
- `attempts` — сколько раз уже пытались чинить JSON;
- `error` — критическая ошибка узла/API.

Это важное отличие: в сложном пайплайне граф должен хранить не только данные, но и **состояние процесса коррекции**.

---

## 5. Как работает граф в `example_parsing_1.py`

Граф собирается так:

`START -> ocr_node -> json_node -> validate_node -> (retry -> json_node) / (done -> END)`

### 5.1. `ocr_node`

Этот узел:

- вызывает `meta-llama/llama-4-scout-17b-16e-instruct` через Groq;
- просит модель извлечь документ максимально полно;
- отдельно говорит сохранять таблицы в формате Markdown;
- возвращает результат в `ocr_text`.

#### Почему это отдельный узел

Потому что OCR и структурирование JSON — это две разные задачи:

- первая требует “видеть” картинку и внимательно читать;
- вторая требует логически разложить данные по схеме.

Разделение повышает качество и делает диагностику проще.

---

### 5.2. `json_node`

Этот узел:

- получает `ocr_text`;
- при необходимости сокращает его через `_shrink_ocr_text_for_json_node()`;
- строит промпт для JSON-модели;
- отправляет текст во вторую модель;
- сохраняет сырой ответ в `raw_json_text`.

#### Зачем сжимается OCR-текст

Большие документы легко упираются в лимиты токенов.  
Поэтому перед отправкой во 2-й узел текст сокращается:

- сохраняется таблица;
- сохраняются ключевые строки;
- отбрасывается второстепенный шум.

Это снижает риск 413 / TPM limit и помогает модели сконцентрироваться на важном.

---

### 5.3. `validate_node`

Это самый важный узел для качества.

Он не генерирует, а **проверяет**.

Проверка идёт в несколько слоёв.

#### Проверка 1. JSON вообще парсится

Сначала код делает:

- `json.loads(raw)`

Если модель вернула нормальный JSON — отлично.

Если нет, включается fallback:

- `_extract_json(raw)`

#### Почему нужен `_extract_json()`

Потому что модели иногда возвращают:

- JSON внутри markdown;
- JSON с пояснениями до/после;
- “почти JSON”, где валидный объект спрятан в тексте.

Это не идеальный путь, но как fallback он полезен.

#### Проверка 2. JSON соответствует схеме

Далее идёт:

- `DocumentSchema.model_validate(...)`

То есть проверяется:

- есть ли обязательные поля;
- правильные ли типы;
- корректно ли приводятся числа.

#### Проверка 3. Полнота списка `items`

Функция `_count_ocr_table_rows(ocr_text)` считает количество строк в OCR markdown-таблице.

Потом сравнивается:

- `expected_rows` — сколько строк было видно в OCR;
- `actual_items` — сколько элементов реально попало в JSON.

Если `actual_items < expected_rows`, значит модель “обрезала” список позиций.

#### Почему выбрана именно такая проверка

Это очень практичная эвристика:

- она дешёвая;
- не требует сложного парсинга таблицы;
- хорошо ловит типичный сбой LLM: “вижу 28 строк, но верну 3”.

#### Проверка 4. Сходится ли итог по сумме

Считается:

- `sum(item.amount_with_vat)` по всем позициям;
- сравнивается с `total_amount`.

Если разница слишком большая, значит:

- потеряны строки,
- либо неправильно распознаны суммы.

#### Почему выбрана именно эта проверка

Это лучший быстрый бизнес-индикатор того, что JSON неполный или арифметически неверный.

Она не гарантирует абсолютную правильность, но хорошо отлавливает реальные ошибки.

#### Проверка 5. Пустой ответ модели

Если `raw_json_text` пустой, это сразу считается ошибкой.

Это защищает от кейса, когда модель вызвалась, но реально ничего полезного не вернула.

---

## 6. Почему число попыток коррекции равно именно 3

В коде это задаётся константой:

- `MAX_JSON_FIX_ATTEMPTS = 3`

### Почему не 1

Одна попытка коррекции часто недостаточна:

- первая генерация может быть неполной;
- вторая, получив `validation_error`, уже понимает, что именно исправлять.

### Почему не 10

Слишком много попыток плохо по нескольким причинам:

- растёт стоимость;
- растёт время обработки;
- если модель “застряла” в одном и том же паттерне ошибки, дополнительные попытки почти не дают пользы;
- появляется риск бесконечного или почти бесконечного цикла.

### Почему 3 — разумный компромисс

Три попытки дают:

1. первичную генерацию;
2. одну содержательную попытку исправления;
3. ещё одну финальную корректировку, если ошибка была составной.

На практике этого обычно достаточно, чтобы:

- исправить формат JSON;
- дозаполнить пропущенные позиции;
- скорректировать суммы.

После третьей попытки вероятность того, что модель внезапно “прозреет”, уже заметно ниже.  
Поэтому 3 — это баланс между:

- качеством,
- стоимостью,
- временем,
- контролируемостью пайплайна.

---

## 7. Почему выбраны именно такие способы проверки JSON

Проверки в `example_parsing_1.py` выбраны не случайно.

### 7.1. `json.loads(...)`

Это минимальная проверка синтаксиса.  
Нельзя работать с данными дальше, пока ответ вообще не является JSON.

### 7.2. `DocumentSchema.model_validate(...)`

Это строгая контрактная проверка:

- проект всегда знает, какой именно JSON он ожидает;
- downstream-код может опираться на схему;
- ошибки ловятся рано.

### 7.3. Подсчёт строк таблицы

Эта проверка отвечает не за синтаксис, а за **полноту данных**.  
Даже идеально валидный JSON бесполезен, если из 28 строк модели попали только 4.

### 7.4. Проверка итоговой суммы

Это бизнес-проверка, а не техническая.  
Она помогает ловить случаи, когда JSON формально корректный, но по сути неверный.

### 7.5. `field_validator`

Это защита от OCR-форматов чисел.

Без неё проект бы тратил попытки ретрая на проблему, которую проще решить кодом.

Именно поэтому часть “ошибок” обрабатывается не моделью, а системой валидации.

---

## 8. Почему не вся валидация делается только моделью

Потому что модель не должна быть единственным судьёй собственного результата.

Если дать второй модели самой решать, “валиден ли JSON”, она часто:

- подтвердит ошибочный результат;
- снова вернёт тот же неполный список;
- не заметит структурные проблемы.

Поэтому архитектура разделяет ответственность:

- модель **генерирует**;
- код **проверяет**;
- модель получает обратную связь и **перегенерирует**.

Это более надёжно, чем доверять всё одной и той же LLM.

## 9. Сравнение файлов

### `code/document_parser_agent.py`

Подходит, если нужен:

- простой пример;
- минимальный пайплайн;
- один вызов одной модели;
- минимум логики.

### `code/example_parsing_1.py`

Подходит, если нужен:

- более устойчивый пайплайн;
- сложные документы;
- контроль полноты таблиц;
- перепроверка итогов;
- механизм самоисправления.

---

## 10. Что важно помнить при развитии проекта

Если проект будет расти дальше, самые важные точки расширения:

- промпт `ocr_node`;
- промпт `json_node`;
- проверки в `validate_node`;
- правила преобразования чисел;
- стратегия сокращения OCR-текста;
- лимит `MAX_JSON_FIX_ATTEMPTS`.

Именно эти места сильнее всего влияют на качество, стоимость и скорость пайплайна.

## 11. Короткий вывод

`code/document_parser_agent.py` — это базовый “одна модель делает всё” пайплайн.  
`code/example_parsing_1.py` — это более зрелая архитектура:

- отдельный OCR;
- отдельное построение JSON;
- отдельная проверка качества;
- повторные попытки исправления;
- защита от типичных ошибок OCR и LLM.

Число повторных попыток выбрано равным **3**, потому что это практический компромисс между качеством и стоимостью.  
Проверки JSON выбраны так, чтобы ловить не только “синтаксические” ошибки, но и реальные бизнес-проблемы: потерю строк, неправильные суммы и невалидные форматы чисел.
